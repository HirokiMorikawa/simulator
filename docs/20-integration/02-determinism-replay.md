# 横断 02. 決定論・再現性 — 固定 dt・状態ハッシュ・リプレイ

crate: `sim-core` / `sim-world`。「検証して遊ぶ」の土台: 同じ条件なら**ビット単位で**同じ結果。

## 1. なぜ決定論が第一級要件か

- **検証**: 解析解との比較・回帰テストは、実行ごとに結果が揺れたら成立しない。
- **遊び**: 「さっきの現象をもう一度」「パラメータだけ変えて比較」が再現性なしでは不可能。
- **デバッグ**: バグの再現が保証される。
- **共有**: シナリオ+シードだけで結果を他人と共有できる(動画不要)。

## 2. 決定論の敵と対策(規約)

| 非決定性の源 | 対策 |
|---|---|
| 可変タイムステップ | 固定 $\Delta t$ のみ。描画は補間([00-foundation/04](../00-foundation/04-architecture.md) §1.3) |
| OS 時計・環境 | コアで `std::time` 等を禁止(lint、[00-foundation/05](../00-foundation/05-rust-wasm-platform.md) §6) |
| 乱数 | SimRng のみ・ストリーム分割([01-math/04](../01-math/04-random.md)) |
| HashMap 反復順序 | 反復するコレクションは Vec / BTreeMap のみ(lint) |
| 並列リダクション順 | チャンク固定・順序固定結合(P3 以降、CI で並列=逐次を照合) |
| 浮動小数の演算順 | 単一バイナリ内では Rust の f64 は決定的。クロスプラットフォーム差は §5 |
| 反復ソルバの順序依存 | 拘束・粒子・セルの処理順を挿入順 ID で固定 |
| イベント順序 | EventQueue は (step, 発生源ID, 種別) でソートして配送 |
| 適応的アルゴリズム | sub-step 数・反復数は状態からの決定的算出のみ(壁時計ベースの打ち切り禁止) |

## 3. 状態ハッシュ

```rust
pub struct StateHasher { /* FNV-1a 64bit ストリーミング */ }
impl World {
    /// 全ドメインの全状態 (f64 のビットパターン) を決定的順序でハッシュ
    pub fn state_hash(&self) -> u64;
}
```

- f64 は `to_bits()` でハッシュ(±0.0 は +0.0 に正規化。NaN が出たら即診断イベント)。
- 順序: ドメイン登録順 → 各ドメイン内は ID 昇順 → フィールド定義順。
- 用途: 決定論 CI テスト(同一シナリオ 2 回 → 一致)、リプレイ照合、UI 常時表示
  (「さっきと同じ状態か」をユーザーが一目で確認)。
- 高頻度呼び出し用に「ダイジェスト版」(全剛体の位置のみ等)も提供(フルハッシュは
  デバッグ・チェックポイントで)。

## 4. シナリオとリプレイ

### 4.1 シナリオ(初期条件の記述)

```rust
pub struct Scenario {
    pub name: String,
    pub seed: u64,
    pub world_options: WorldOptions,        // dt, 重力, 環境温度, 大気...
    pub bodies: Vec<RigidBodyDesc>,
    pub joints: Vec<JointDesc>,
    pub fluids: Vec<FluidDesc>, pub circuits: Vec<CircuitDesc>, /* … 各ドメイン */
    pub couplings: Vec<CouplingDesc>,       // 排他検査は validator ([20-integration/01] §5)
}
```

- 直列化は JSON(人間可読・手編集可能・共有しやすい)。数値は f64 を損なわない表現
  (整数ビットまたは 17 桁)で書く。
- シーン内の乱数配置(散乱など)もシードから決定的に生成。

### 4.2 リプレイ

```rust
pub struct Replay {
    pub scenario: Scenario,
    pub inputs: Vec<(u64 /*step*/, Command)>,   // ユーザー操作列 (決定的に再適用)
    pub checkpoints: Vec<(u64, u64)>,           // (step, state_hash) 1秒ごと
}
```

- **入力もステップ番号に量子化**して記録 → シナリオ+入力列だけで完全再生。
- 再生時に checkpoints と実ハッシュを照合 — 不一致は「エンジンのバージョン差 or バグ」を
  即検出(バージョン番号を Replay に埋める)。
- スナップショット(全状態の直列化)は「途中から再開」「巻き戻し」用。
  スナップショット→再開の結果が連続実行と一致することを CI で検証。

## 5. クロスプラットフォーム決定論(範囲の宣言)

- **保証**: 同一バイナリ(同一 wasm モジュール / 同一ネイティブビルド)内での完全一致。
- **目標(ベストエフォート)**: wasm ⇔ ネイティブ、ブラウザ間。IEEE 754 準拠の
  基本演算(+−×÷√)は一致するが、超越関数(sin, exp, pow)は libm 実装差がありうる。
  対策: コアの超越関数を自前実装(必要精度の多項式近似、`sim-math::det_math`)に置換し、
  全プラットフォームでビット一致させる。FMA の自動生成は無効化(コンパイラフラグ)。
- CI で wasm 実行(node)とネイティブのハッシュ照合を常設し、一致範囲を実測で文書化する。

## 6. 検証

- 同一シナリオ 2 回実行 → 全チェックポイントハッシュ一致(全デモシーンで CI 必須)。
- スナップショット再開 = 連続実行(1000 step 後のハッシュ一致)。
- 入力リプレイ: 記録セッションと再生セッションの一致。
- カオス系(二重振り子・三体)でこそ一致を確認(丸め 1 bit の差も指数増幅されるため、
  最も感度の高い決定論テストになる)。
