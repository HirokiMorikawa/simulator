# 横断 03. エンティティ層 — 物・人・乗り物・生物の物理契約

crate: 将来の `sim-entity`(物理エンジンの**クライアント**)。本文書は「エンティティが
物理エンジンに何を要求するか」の契約を定義し、エンジン API([04-world-api.md](04-world-api.md))の
要件源泉となる。エンティティ自身の詳細設計(AI・行動)は本設計書群のスコープ外。

## 1. 設計原則

1. **エンティティは物理の上に立つ、中には入らない**: World 公開 API のみ使用。
   必要な機能が API に無ければ API を拡張する([00-foundation/04](../00-foundation/04-architecture.md) §4)。
2. **物理を欺かない**: エンティティの運動はすべて力・トルク・モーター経由。
   位置の直接テレポートは編集操作(シミュレーション外)としてのみ許す。
   → 人が壁を押せば反作用で押し返される、が自動で成立する。
3. **構成的**: すべてのエンティティは「剛体 + ジョイント + アクチュエータ + センサー + 制御」の
   組み合わせで表現する。特別扱いのエンティティ型をエンジンに作らない。

## 2. 物(Objects)

- = 剛体(+ 熱ノード + 材質)。既に全ドメイン文書でカバー。
- 複合物(家具・道具): Compound 形状 or Fixed ジョイント結合。破壊 = ジョイント破断
  ([10-mechanics/05](../10-mechanics/05-joints-constraints.md) §3 breaking_impulse)。

## 3. 人・生物(Humans / Creatures)

### 3.1 身体 = 関節骨格(articulated body)

```
人体 (~15 剛体): 頭・胴2・骨盤・上腕2・前腕2・手2・大腿2・下腿2・足2
関節: Ball (肩・股) / Hinge (肘・膝) / 制限付き Ball (首・脊椎・手首・足首)
寸法・質量分布: 人体計測データ (Winter, "Biomechanics and Motor Control of Human Movement"
の体節質量比: 頭 8.1%, 胴 49.7%, 大腿 10%, 下腿 4.65%, ...) を身長・体重でスケール
```

- **ラグドール**(受動)が Phase 3 の到達点: 階段を転げ落ちる、椅子に崩れ込む。
- **アクティブ制御**(Phase 4+): 各関節のモーター(PD サーボ、
  [10-mechanics/05](../10-mechanics/05-joints-constraints.md) §4.5)に目標姿勢を与える
  コントローラ。バランス制御(倒立振子近似)・歩行(ZMP or 学習)はエンティティ層の課題。
  物理エンジン側の要件は「モーター行のトルク上限・仕事の記帳」のみ(済)。

### 3.2 エンジンへの要求(センサー)

| センサー | 対応 API |
|---|---|
| 足裏接地・荷重 | 接触イベント + `Constraint::impulse()` |
| 関節角・角速度 | ジョイント状態クエリ |
| 前庭感覚(傾き・加速度) | 剛体状態(姿勢・加速度履歴) |
| 視覚(将来) | ray-cast バッチ / 光学ドメイン |
| 温度感覚 | ThermalNode 温度 |

### 3.3 生物の生理(熱との結合)

- 代謝 = 定常熱源(人体 ~100 W、運動時 ~500 W)を胴の ThermalNode に。
- 体温調節(発汗 = 蒸発冷却 [12-thermal/03](../12-thermal/03-phase-change.md) §2.3、ふるえ = 熱源増)は
  エンティティ制御の例題。低体温・熱中症の境界(体温 35/40 °C)を持つ生存モデルが
  「環境の物理が生物に意味を持つ」デモになる。
- 浮力・水泳: 人体平均密度 ≈ 1010 kg/m³(肺の空気で ±)— 息で浮き沈みするデモ。

## 4. 乗り物(Vehicles)

### 4.1 陸上車両(Phase 4)

```
車体 = 剛体 (箱 or 複合)
車輪 = WheelJoint × 4:
  - サスペンション: Slider + ソフト拘束 (ばね定数・減衰は実車パラメータ)
  - 駆動: ヒンジモーター (トルク源: モデルエンジン曲線 or DCモーター [13-em/05])
  - 操舵: ヨー軸ヒンジ (前輪)
タイヤ力: 接触摩擦の拡張。縦力: スリップ比 κ の関数、横力: スリップ角 α の関数
  (簡易 Pacejka Magic Formula: F = D·sin(C·atan(B·s)) — 係数は §9)
```

- タイヤモデルは接触ソルバの摩擦上限 $\mu N$ を**スリップ依存**に置き換える形で実装
  ([10-mechanics/04](../10-mechanics/04-friction.md) の拡張点)。
- 検証遊び: 制動距離 vs 路面(乾燥/濡れ/氷)、遠心力とロールオーバー閾値、
  エンジン出力と登坂限界。

### 4.2 船・飛行機(Phase 4+)

- 船 = 剛体 + 浮力(部分浸水 [11-fluid/04](../11-fluid/04-free-surface-buoyancy.md)、復原性)+
  水抗力 + スクリュー推力(単純推力 or モーター結合)。
- 飛行機 = 剛体 + パネル揚力/抗力([11-fluid/05](../11-fluid/05-aero-hydrodynamics.md) §2.2、
  主翼・尾翼・舵面)+ プロペラ推力。失速・トリム・重心位置の遊びが成立する最小構成。

### 4.3 動力の物理的整合

乗り物の動力は必ずエネルギー源に接続する: 電池→モーター(記帳済み)、
燃料→熱機関(簡易オットーサイクル効率 × 燃料流量、Phase 5)。
「無から湧くトルク」を作らない — 燃費・航続距離が創発する。

## 5. スポーンと編集(シミュレーションの外側)

- エンティティ配置・パラメータ変更は「編集モード」= 新しいシナリオの生成として扱い、
  実行中の連続性(決定論)と区別する。実行中の介入(押す・持つ)は
  Command(力・モーター指令)として記録されリプレイ可能([02-determinism-replay.md](02-determinism-replay.md) §4.2)。

## 6. 他ドメインとの結合

エンティティは結合を新設しない — 既存の結合(モーター・熱・浮力…)の消費者である。
これが成立しているかが結合行列の完全性のチェックになる。

## 7. 検証(エンティティ契約の受け入れテスト)

- ラグドール落下: エネルギー単調減少・関節可動域の維持・貫入なし。
- 立位保持(P4): PD 制御のみで 60 s 直立(外乱 10 N プッシュに復元)。
- 車両: 制動距離 $v^2/(2\mu g)$ ± 10%(タイヤモデル簡易化の範囲)、
  定常円旋回の横加速度と遠心力の釣り合い。
- 船: 復原モーメント曲線(傾斜角 vs モーメント)がメタセンタ計算と一致。
- 人の浮沈: 肺気量での浮き沈み(密度 ±2% の感度)。

## 8. 実装フェーズ対応

P3: ラグドール(人形)。P4: 車両・アクティブ制御の基礎・船。P5+: 歩行・飛行機・生理モデル拡張。
