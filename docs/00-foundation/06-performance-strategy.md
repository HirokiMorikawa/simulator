# 06. 性能戦略 — アルゴリズム・並列化・データ局所性・GPU

crate: 横断(全ドメイン)。「検証して遊ぶ」に必要な速度を、**正しさと決定論を壊さずに**達成するための方針。
[05-rust-wasm-platform.md](05-rust-wasm-platform.md) の性能予算を満たす手段を定義する。

## 1. 原則

1. **正しさ → 決定論 → 速度の順で確立する**。スカラー・シングルスレッドの参照実装を常に残し、
   最適化版はそれと結果一致(決定論の許容内)を CI で保証する([20-integration/02](../20-integration/02-determinism-replay.md))。
2. **最適化は実装(Phase B)と同時に、ドメインごとに適用する**。後付けの全面改修を避け、
   各ドメインの最初のバージョンから SoA・アルゴリズム選択を織り込む([22-roadmap/01](../22-roadmap/01-phases.md))。
3. **精度を静かに落とさない**。速度不足は規模縮小・スロー再生・オフライン計算で吸収する
   ([01-vision.md](01-vision.md) §5)。近似を増やす場合は明示する。
4. **測ってから最適化する**: `criterion`(ネイティブ)の常設ベンチで回帰を検知し、
   プロファイルでホットスポットを特定してから手を入れる。

## 2. 三本柱

### 2.1 アルゴリズムの上位化(計算量オーダーを下げる)

最も効果が大きい。各ドメインの標準的な高速アルゴリズムを最初から採用する:

| ドメイン | 素朴 | 上位アルゴリズム | オーダー |
|---|---|---|---|
| 衝突 broadphase | 総当たり $O(n^2)$ | Sweep-and-Prune → BVH(動的 AABB木) | $O(n\log n)$ |
| 衝突 narrowphase | — | GJK/EPA(凸)、SAT(箱) | 定数 |
| 圧力 Poisson(流体) | Jacobi | PCG → **マルチグリッド前処理** | ~$O(n)$ |
| SPH 近傍探索 | 総当たり | 空間ハッシュ + Morton 順 | $O(n)$ |
| 熱・拡散 | 陽的反復 | 陰的 + マルチグリッド | 無条件安定 |
| N体重力 | 総当たり $O(n^2)$ | **Barnes-Hut** → FMM | $O(n\log n)$ / $O(n)$ |
| 回路 | 毎回フル LU | 疎 LU + トポロジ不変時の分解キャッシュ・選択的再計算 | — |
| FDTD | 一様格子 | サブグリッド(局所細分化) | 定数倍改善 |
| パストレ | 一様サンプリング | 重要度サンプリング + BVH + NEE | 分散低減 |

原則: **O を下げる余地があるなら定数倍最適化より先に取り組む**。

### 2.2 SIMD / 並列化

- **並列化(データ並列)**: ネイティブ `rayon`、ブラウザ `wasm-bindgen-rayon`
  (SharedArrayBuffer + COOP/COEP ヘッダが必要)。適用単位はドメインごとに規定(§4 各 §10):
  流体格子のスライス分割、SPH の近傍バケット、broadphase のペア、N体のツリー枝、
  接触ソルバの**接触島**(独立な島は並列、島内は逐次)。
- **SIMD**: `std::simd`(ネイティブ)/ `wasm simd128`(128bit = f64×2 or f32×4)。
  対象はホットカーネル: 格子ステンシル、SPH カーネル評価、ベクトル演算のバッチ。
- **決定論の制約(必須)**:
  - 並列リダクション(総和・最小値)は**チャンク分割を固定し、チャンク順に逐次結合**する
    (浮動小数加算は非結合的なので、スレッド完了順に足すと結果が変わる)。
  - 反復ソルバの並列化は赤黒順序付け等で**決定的な更新順序**を保つ。
  - スレッド数に依存しない結果を CI で検証(1 スレッド = N スレッドのハッシュ一致)。
  - PRNG は粒子/チャンクごとに固定ストリーム([01-math/04](../01-math/04-random.md) §2)。
- 段階導入: Phase B で各ドメイン実装時にスカラー版を先に通し、直後に SIMD/並列版を
  参照一致テスト付きで追加する。

### 2.3 データ局所性(メモリレイアウト)

- **SoA(Structure of Arrays)**: 全ドメインの既定([00-foundation/04](../00-foundation/04-architecture.md) §3)。
  属性ごとの連続配列でキャッシュ効率と SIMD・WASM 転送を両立。
- **キャッシュブロッキング/タイリング**: 格子系(流体・熱・FDTD)のステンシル走査を
  L1/L2 に収まるタイル単位で(例 8³ ブロック)処理し、再利用を最大化。
- **空間局所性ソート**: 粒子(SPH・分子・ブラウン)を **Morton(Z オーダー)** で定期ソートし、
  近傍アクセスをキャッシュ内に収める。ソートは安定(インデックス順タイブレーク)で決定的。
- **ホット/コールド分離**: 毎ステップ触るフィールド(位置・速度)と稀にしか触らないフィールド
  (材質 ID・デバッグ情報)を別配列に分け、ホットループのキャッシュライン充填率を上げる。

## 3. GPU(WebGPU)— 第一級設計・CPU 優先

大規模な格子/粒子ドメイン(格子流体・FDTD・SPH・N体・パストレ)の高速化手段として
**設計に組み込むが、既定は CPU**。GPU は性能が要るシーンで有効化する差し替え可能なオプション。

### 3.1 隔離設計

- 各ドメインの解法トレイト(`FluidSolver` / `FdtdSolver` / `SphSolver` / `NBodySolver` / `Renderer`)の
  **別実装**として GPU バックエンドを置く。ドメインの結合インターフェース(力の照会・境界条件・
  状態読み出し)は CPU/GPU で不変。World・結合層は GPU の存在を知らない。
- API: `wgpu`(ネイティブ・ブラウザ両対応)。シェーダは WGSL。CPU 参照実装は常に残す。

### 3.2 決定論と検証

- GPU の並列リダクションも**順序固定**(セグメント化リダクション + 決定的結合順)を課す。
  浮動小数の再現性が GPU ドライバ間で崩れうるため、**GPU モードの決定論保証は「同一デバイス+
  同一ドライバ」に限定**して宣言する([20-integration/02](../20-integration/02-determinism-replay.md) §5 と同じ範囲限定の考え方)。
- CPU 参照実装との結果一致を許容誤差内(物理検証と同じ閾値)で CI 照合。厳密ビット一致は
  GPU では要求しない(できない)— この妥協を明示する。決定論が絶対要件のシーン
  (リプレイ検証・回帰)は CPU で実行する規約。

### 3.3 転送・パイプライン

- ホスト⇔デバイス転送を最小化: シミュレーション状態を GPU 常駐にし、描画バッファは
  GPU 内で共有(WebGPU の compute→render 連携)。CPU が読むのは観測値・低頻度クエリのみ。
- 非同期パイプライン: compute パスを次フレームとオーバーラップ。ただし決定論のため
  ステップ境界は同期(状態ハッシュ・結合の一貫性)。

## 4. ドメイン別性能プロファイルのテンプレート

各ドメイン文書は末尾に「§10 性能プロファイル」を持ち、以下を記述する:

```
- ホットスポット: (プロファイルで支配的な計算)
- 目標アルゴリズムとオーダー: (§2.1 の該当行)
- SoA レイアウト: (主要配列とホット/コールド分離)
- 並列化単位: (rayon の分割対象)
- SIMD 対象カーネル: (simd128 化する内側ループ)
- GPU 適性: (高/中/低 と理由。オフロード対象か)
- ベンチ: (criterion の代表シーンと目標時間)
```

## 5. 検証

- **回帰ベンチ**: 各ドメインの代表シーンを `criterion` で常設、CI でしきい値超過を検知。
- **決定論 CI**: スカラー=SIMD=並列(=GPU は許容誤差内)の結果一致を全ドメインで。
- **性能予算 CI**: [05-rust-wasm-platform.md](05-rust-wasm-platform.md) §5 の予算表を代表シーンで実測し、
  超過を警告(fail ではなく可視化 — ハード保証でなく監視)。
- **スケーリング測定**: 規模 n を変えて実測オーダーが理論通りか($n^2$ でないか)を確認。
