対象: PR #1「物理エンジン設計書一式」head `338c70d`(53 文書 + 初回レビュー、約 6,700 行)。

位置づけ: 初回レビュー([pr1-design-review.md](pr1-design-review.md)、head `a775522` 時点)の後、
設計文書には一切の改訂が入っていない(以後のコミットはレビューの追加とそのマージのみ)。

レビュー観点は前回と同じ — プロジェクトの目的(現実の写像を検証して遊ぶシミュレータ)に照らし、

## 総評

初回レビューの指摘(A〜E および障害予測)を本文書群と突き合わせて再検証した結果、
**全項目が現行 head でも成立し、いずれも未対応**である。結論「この設計・この工程では到達しない」
は維持される。

さらに第 2 回の精査で、初回が指摘しなかった問題が新たに見つかった。中でも重大なのは、
物理の中身ではなく**設計フェーズのプロセスが既に自壊している**こと(R-1)と、
「テスト表 = 実装仕様」という本設計の看板に対する**具体的な反例**(R-2)である。

## R-1. プロセスの自己矛盾 — 「到達しない」と自認したまま承認ゲートへ(最重大・新規)

PR #1 は、自らの内部に「この設計・この工程では到達しない」と結論する正式レビュー
(PR #2 としてマージ済み)を取り込んだまま、設計・ロードマップを **1 行も改訂せずに**
レビュー承認 = 実装開始のゲート([00-foundation/01-vision.md](../00-foundation/01-vision.md) §4)を求めている。

本設計書群は自らに次の規約を課している:

- 「実装が設計から乖離したら**設計書を先に改訂**する」([22-roadmap/01-phases.md](../22-roadmap/01-phases.md) 横断ルール 5)
- 「新要望は**まず本ロードマップの改訂 PR** にする」(同 リスク表)

この「指摘 → 文書改訂」のループは、最初の適用機会 — 初回レビューの最重大指摘(規模見積り不在・
工程の構造欠陥)— で既に回っていない。設計しか存在しない、最も改訂コストが低い現段階で
守られない規約が、52 文書と実装コードの二重メンテが要る実装フェーズで守られる見込みは低い。

**このまま承認した場合の帰結**: 「到達不能と自認した工程」が正式化され、以後
「設計書先行改訂」規約は最初から実効性のない飾りとして出発する。初回レビュー A-2 が予測した
「設計書群が現実を写さない文書に転落する」過程は、実装開始前から始まっていることになる。

## R-2. 検証設計の実装不能の実例 — E2 サイクロトロンテスト(新規・具体)

解析解テスト表([21-verification/01-analytic-tests.md](../21-verification/01-analytic-tests.md))は
「実装フェーズの cargo test がそのまま実装すべきテスト仕様」を掲げるが、E2 は設計内の道具では
合格できない。

- **E2 の要求**: サイクロトロン運動で「$r = mv/(qB)$、**速さ一定**」、許容 rel 0.5% / **abs 1e-9**。
- **積分器カタログ**([01-math/03-integrators.md](../01-math/03-integrators.md))の全所帯:
  semi-implicit Euler(剛体の既定)、velocity Verlet(「$\mathbf{a}$ が $\mathbf{x}$ のみに依存する
  保存系」向けと自己宣言 — 速度依存のローレンツ力には適用外)、RK4(無衝突の基準値生成専用・
  非シンプレクティック)。
- 磁場中の荷電粒子に semi-implicit Euler を使うと
  $\mathbf{v}_{n+1} = \mathbf{v}_n + \Delta t\,\frac{q}{m}\mathbf{v}_n \times \mathbf{B}$ となり、
  $|\mathbf{v}_{n+1}| = |\mathbf{v}_n|\sqrt{1 + (\omega\Delta t)^2}$ — 速さは**毎ステップ系統的に増大**する。
  abs 1e-9 の速さ保存は sub-step をいくら積んでも構造的に届かない。磁場中の回転を厳密に
  ノルム保存する積分器(Boris pusher 等)が必要だが、カタログに存在せず、
  E2 を担う [13-electromagnetism/01](../13-electromagnetism/01-electrostatics-magnetostatics.md)・
  [05](../13-electromagnetism/05-em-mechanics-coupling.md) にも積分法の指定がない。

**予測される障害**: Phase A で E2 を書く時点か、P4(LorentzForce 結合)で Green にする時点で、
最初期の math ウェーブ成果物(`Integrator` トレイト・積分器カタログ)への手戻りが発生する。
一事が万事で、約 100 本のテスト表には「書いてみたら/実装してみたら仕様側が壊れている」ものが
他にも混ざっている可能性が高く(R-5, R-6 も同種)、「テスト表は転記するだけ」という
Phase A の前提を崩す。

## R-3. エンティティ層の受け入れ基準に研究級課題が混入(新規)

[20-integration/03-entity-layer.md](../20-integration/03-entity-layer.md) は
「バランス制御(倒立振子近似)・歩行(ZMP or 学習)は**エンティティ層の課題**」として
エンジン設計のスコープ外に送っている(§3.1)。ところが同じ文書の受け入れテスト(§7)は
「**立位保持(P4): PD 制御のみで 60 s 直立(外乱 10 N プッシュに復元)**」を数値固定の
合格基準にしており、ロードマップの P4 にも「アクティブ制御の基礎」として載っている。

15 剛体の関節骨格を関節 PD だけで 60 秒間立たせ外乱から復元させるのは、二足バランス制御という
それ自体が研究テーマの課題であり、「スコープ外に送った当の問題がフェーズ完了条件に入っている」
自己矛盾である。しかも P4 は回路・光学・WCSPH・車両(Pacejka)・ブラウン運動に加えて
この立位制御を並行する、全ウェーブ中の最重量区間になっている。

**予測される障害**: P4 の完了条件が満たせず、(a) 基準の黙殺(Definition of Done の形骸化が
ここから始まる)か、(b) 制御工学のサブプロジェクト化による P4 の長期停滞、の二択になる。

## R-4. 性能予算表は加算で破綻している(新規)

[00-foundation/05-rust-wasm-platform.md](../00-foundation/05-rust-wasm-platform.md) §5 は
「フレーム 16.6 ms のうち物理に 10 ms」と宣言する一方、ワークロード別予算の合計は
剛体 3 + 格子流体 4 + SPH 4 + 気体分子 2 + FDTD 5 + 天体 1 + その他 1 ≈ **20 ms** であり、
**どのワークロードが同時に生きてよいかの構成規則がどこにもない**。

単一ドメインのデモなら個別予算で足りるが、本設計の売りである多ドメイン統合シナリオ —
D18 氷と飲み物(剛体+流体+熱+相変化)、D37 再突入(天体+流体+熱+剛体)、
Phase C の統合テスト群 — での配分は未定義である。さらに性能予算 CI は
「fail ではなく可視化 — ハード保証でなく監視」([00-foundation/06-performance-strategy.md](../00-foundation/06-performance-strategy.md) §5)
と最初から拘束力を放棄しており、予算表は設計上の約束として機能しない。

**予測される障害**: 統合シナリオが 60 fps を割った時点で「規模を落とす/スロー再生」の判断基準が
なく、シーンごとの場当たり調整になる。予算監視は警告の常態化 → 無視、の経路をたどる。

## R-5. S7/S8(イジング)の合格基準と実装順の食い違い(新規)

S7/S8 は「L=256 で $T_c$ rel 2%・Onsager 自発磁化 rel 2%」を **P5 の Green 条件**とするが、
臨界減速対策の Wolff クラスタ法は「オプション実装」「Wolff・3D は**その後**」
([15-statistical/04-monte-carlo.md](../15-statistical/04-monte-carlo.md) §)と P5 の後に送られている。
単スピン Metropolis の臨界域の相関時間は $\tau \sim L^z$($z \approx 2.17$)で増大し、
L=256 の $T_c$ 近傍を平衡化して帯磁率ピークを分解する計算は CI テストとして成立する長さに
収まらない。**テストを Green にする手段が、そのテストより後の工程に置かれている。**

## R-6. F11 カルマン渦の「検証モード」は渦が出ない恐れ(新規・小)

F11(ストローハル数 $St \approx 0.2$、rel 20%)の検証モードは渦度強化オフ
([11-fluid/02-eulerian-grid.md](../11-fluid/02-eulerian-grid.md) §4.5: 既定 0)で走らせる規約だが、
semi-Lagrangian 移流の強い数値粘性の下、64³ 格子で渦離脱が自発的に立ち上がるかは
それ自体が不確実である。合格基準が「非物理項を切った状態」を要求し、その状態では
現象自体が消えうる — rel 20% の緩さでも Red に固着するリスクがある。

## このまま進めた場合の障害予測(時系列、初回版の更新)

0. **設計フェーズ(現在)**: 「レビュー指摘 → 設計改訂」のループが回っていない(R-1)。
   承認と同時に、横断ルール 5 は実績ゼロの規約として実装フェーズへ持ち込まれる。
1. **Phase A**: 数週間〜数ヶ月、緑ゼロのテストを書き続ける(初回 B-1)。加えて E2 型の
   「設計内の道具では書けない/合格できないテスト」(R-2, R-5, R-6)が次々に見つかり、
   テスト先行の建前がフェーズ内で崩れる。Phase B 開始直後に API 変更でテストの大規模書き直し。
2. **P1–P3**: 最初の泥沼は物理ではなく決定論 CI(並列=逐次一致・det_math 整備)。
3. **P4**: 専門性の壁(MNA 収束・SPH 安定化)に加え、立位制御(R-3)で完了条件が満たせず、
   Definition of Done の黙殺かサブプロジェクト化かの二択に陥る。統合シナリオで性能予算の
   構成規則不在(R-4)が表面化する。
4. **Pα**: floating origin が力学コアのデータ設計改修を要求(初回 C-3)。
   時間加速 × 固定 dt の矛盾が表面化(初回 C-4)。
5. **Phase D**: 到達しない。
6. **全期間**: 決定論・保存則・ベンチ CI の維持コストが実装を上回り、ゲートの選択的無効化が
   進み、中核価値の決定論が静かに劣化する。性能予算監視は警告常態化の後に無視される。

## 結語

設計文書の物理的な質は初回レビューの評価どおり高い。しかし第 2 回として問われるべきは
「初回の指摘に工程がどう応答したか」であり、答えは「応答していない」である。
設計の中身の欠陥(R-2〜R-6 のような具体的な仕様バグを含む)は個別に直せるが、
指摘が文書改訂につながらないプロセス(R-1)の下では、どれだけレビューを重ねても
設計はゴール到達可能な形に収束しない。到達性の最大のリスクは、もはや設計書の中ではなく
設計書の外 — 改訂ループの不在 — にある。
