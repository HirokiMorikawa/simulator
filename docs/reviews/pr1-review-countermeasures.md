## 1. 総括 — 対策の骨格

両レビューの結論「この設計・この工程では到達しない」の根因は、(1) 全ドメイン同時進行の
工程構造(Phase A 全テスト先行、ユーザー価値の到達点が遠い)、(2) 横断機構の未設計・過大な
不変条件(floating origin、固定 dt×時間加速、ビット決定論)、(3) 仕様として成立していない
テスト行の混入(E2/S7/S8/F11)、の 3 系統に分解できる。対策の骨格は次のとおり。

1. **工程の再編**: Phase A(全テスト先行)を廃止し、テスト先行を各実装ウェーブの内部に局所化する。
   各ウェーブ末に「遊べるデモ」をリリースし、進捗と到達性をデモで計測する。
   寄与の薄い重量級ドメイン(量子・FDTD・イジング・1PN・スペクトルパストレ)は削除せず、
   コアのクリティカルパスから外れた**独立トラック**に再配置する。
2. **前倒し決定**: 後から変えると土台改修になる設計判断(位置表現のフレーム、決定論の水準、
   最小 CCD)を該当ウェーブより前に決定・設計するタスクとして登録する。
3. **仕様の修理**: テスト表・受け入れ基準のうち設計内の道具で合格できない行(E2, S7/S8, F11,
   立位保持)を、積分器カタログの拡充・基準の段階化・CI 階層化で合格可能な形に改訂する。
   併せてテスト表全行の**実装可能性監査**を行い、同種の仕様バグを事前に洗い出す。

## 3. 指摘別の対策詳細

### A. 規模・自己評価

**A-1(工数・期間・体制の記述が無い)** — 決定 2 により人月・期間・体制の見積り節は
新設しない。AI が実装を行う体制では人月は到達性の尺度として機能しないためである。
ただし指摘の本質「そこまでの距離を測る材料が無い」には応える必要があるので、距離の
尺度を人月から**進捗の観測可能性**に置き換える:

- 各ウェーブの完了条件(Green にするテスト番号・合格デモ番号)は現行のまま維持する。
- B-3 の対策(各ウェーブ末に動くデモをリリース)により、「どのウェーブまで到達したか」が
  常に動くソフトウェアで観測できる状態を保つ。
- phases.md の横断ルールに「進捗はウェーブ完了とデモリリースで計測する。長期間デモが
  リリースされないことを停滞のシグナルとして扱う」を追記する。

**A-2(「実装直前レベル」は過大自己評価)** — 自己評価を実態に合わせて修正し、乖離を
前提とした運用に切り替える:

- vision §4 の「実装直前レベル」「転記・翻訳するだけ」を「アルゴリズム選定と骨子の確定
  レベル。実装の難所はウェーブ着手時の実装設計スパイクで詳細化する」に改訂する。
- 各ウェーブの先頭に**実装設計スパイク**(接触マニフォールドの退化ケース、SPH 境界粒子、
  MNA 非線形収束などの難所を、実装前に短い使い捨てコードで検証する工程)を置く。
- 二重メンテ税(52 文書の恒常改訂)対策として、設計書の粒度規約を定める: 設計書が保持する
  のは「何を・なぜ」(アルゴリズム選択、不変条件、インターフェース、合格基準)まで。
  「どう」(実装詳細)はコードとテストが持つ。横断ルール 5 の「設計書を先に改訂」の義務は
  この粒度での乖離(アルゴリズム変更・不変条件変更・API 変更)に限定し、実装詳細レベルの
  差異では発動しない。

### B. 工程(TDD)

**B-1(Phase A は BDUF)** — Phase A(全ドメイン・全テスト先行)を廃止し、テスト先行を
各ウェーブの内部に局所化する:

- 新しい工程: Phase 0(骨格)→ 各ウェーブ(math, P1, …)の内部で「そのウェーブが担う
  テスト表の行をテストコード化 → Red → 実装 → Green」を回す。Red→Green のループが
  ウェーブ単位(週オーダー)で閉じ、「数ヶ月全 Red」の期間が消える。
- 解析解テスト表(analytic-tests)は**仕様カタログ**として全量維持する。変わるのは
  「いつテストコード化するか」だけであり、検証設計の資産価値は失わない。
- analytic-tests §0 の TDD 運用規約を上記に合わせて改訂する。

**B-2(CI ゲート実行時間が未検討)** — CI を 3 階層に分離し、各テストがどの層で走るかを
テスト表の列として明記する:

| 層 | 実行契機 | 内容 | 目標時間 |
|---|---|---|---|
| PR ゲート | 全 PR | 単体・解析解テストの縮約版・決定論(同一シード 2 回ハッシュ一致)・fmt/clippy | 分オーダー |
| ナイトリー | 毎日 | 保存則 residual・収束次数 4 水準・ベンチ回帰・デモシーン決定論 | 1 時間級 |
| 週次/リリース | 週 1 + リリース前 | 長時間級(A2 二体 10⁶ 周、S7/S8 フル L=256、並列=逐次照合の全シーン版) | 制限なし |

- A2 のような 10⁸–10⁹ ステップ級は「縮約版(例: 10³ 周)を PR ゲート、フル版を週次」に
  分離する。縮約版の許容誤差はフル版から外挿して別途定める。
- 「必須ゲート」の語は PR ゲート層のみに使い、上位層は「必須だが非同期」(失敗したら
  翌営業日に最優先で修理する規約)とする。これにより形骸化(選択的無効化)ではなく
  制度としての階層化で実行時間問題を解く。

**B-3(ユーザー価値の到達点が遠すぎる)** — 「遊べる」を各ウェーブの出口に置く:

- 横断ルールに「各ウェーブの完了条件に、そのウェーブのデモ(D 番号)がブラウザで遊べる
  状態でリリースされることを含める」を追加する。P1 完了 = D1–D10 が遊べる、が最初の
  ユーザー価値到達点になる。
- Three.js リアルタイムプレビュー(rendering-architecture §2 の経路 (a))を Phase 0–P1 に
  前倒しする。これは新規スコープではなく、既に設計にある経路 (a) の実装時期の明確化である。
- Phase D(物理正確スペクトルパストレ = 経路 (b))は独立トラック(後述 D)に移す。

### C. 横断機構

**C-1(ビット決定論が最も高価な不変条件)** — 決定 3 により両案を併記する。
**決定期限は P1 着手前**(P3 で並列リダクション、det_math はそれ以前から効くため)。

- **案 1: 決定論の階層化(緩和)**
  - ビット一致を要求する範囲を「同一バイナリ・同一設定・同一スレッド数でのリプレイ再現」に
    限定する。シード・入力列からの再現、状態ハッシュ、リプレイはこの範囲で完全に機能する。
  - 並列=逐次の一致、wasm⇔ネイティブのクロスプラットフォーム一致は**許容誤差ベース**
    (状態ノルム相対誤差の閾値)に格下げする。
  - `det_math` 自前実装と FMA 無効化を撤回する(libm 準拠+コンパイラ既定)。
  - 利点: 全最適化 PR がハッシュ一致テストと戦う開発税が消える。rlibm 級サブプロジェクトが
    消える。FMA 性能を回収できる。/ 欠点: 「並列度を変えても同一結果」という強い再現性を
    失う。バグ報告の再現にスレッド数の記録が要る。
- **案 2: ビット決定論の全面維持**
  - 現行設計(順序固定リダクション・det_math・FMA 無効化)を維持する。
  - 対策は (i) B-2 の CI 階層化(並列=逐次の全シーン照合は週次層へ)、(ii) det_math を
    独立トラックのサブプロジェクトとして明示的にタスク計上(スコープ: sin/cos/exp/pow の
    正確丸め実装または既存 rlibm 系クレートの採用評価)、(iii) FMA 無効化の性能影響を
    Phase 0 でベンチ計測し予算表に反映、の 3 点で開発税を管理可能にする。
  - 利点: 決定論という中核価値が無傷。/ 欠点: 開発税は残る(管理されるだけ)。
- 推奨は案 1(リプレイ・検証・遊びの体験は同一バイナリ内ビット一致で成立し、並列=逐次
  一致が守る価値は開発コストに見合わない)だが、決定は保留し P1 着手前に行う。

**C-2(エネルギー台帳恒等式は恒真式に堕ちる)** — 指摘は正しいと認め、台帳の役割を
再定義する:

- $R_{numerical}$ は「独立に計算する量」ではなく「ステップ前後のエネルギー差分から
  既知の仕事・熱を引いた**残差**」と正直に定義し直す(architecture §1.1.2(2))。
- 台帳の役割を「不明なエネルギー漏れの常時**検出器**」から「残差の**トレンド監視指標**」に
  格下げする: 残差の時系列を記録し、(i) 単調増大・階段状ジャンプなどのパターン検知、
  (ii) シーンあたり累積残差の緩い上限、の 2 つだけを CI 対象にする。項別閾値の手動調整には
  依存しない。
- バグ検出の役割は、解析解が既知のシナリオに対するドメイン別保存則テスト
  ([21-verification/02-conservation-laws.md](../21-verification/02-conservation-laws.md))が
  担うことを明記し、両者の役割分担を architecture §1.4.4 に書く。

**C-3(floating origin が約 15 行)** — 2 段階で対策する:

- **P1 着手前(決定)**: 力学コアの位置表現を決める — 「剛体の位置は所属フレームの
  ローカル座標 f64 で保持し、World は フレーム ID + ローカル座標を正準状態とする」を
  デフォルト案として採否を決定する。これを P1 の `RigidBody` データ設計に最初から
  織り込む(後から変えると土台改修になる、というレビューの指摘どおりの理由で前倒す)。
- **Pα 着手前(詳細設計)**: フレーム階層設計文書を新設し、(i) フレーム跨ぎ時の接触・拘束の
  扱い、(ii) 非慣性項(コリオリ・遠心)の台帳計上、(iii) 跨ぎ判定の決定論(ヒステリシス付き
  閾値・跨ぎイベントの順序固定)、(iv) 跨ぎの単体テストを規定する。scale-ladder §2.2 は
  この文書への参照に置き換える。
- architecture §4 の拡張原則には「フレーム階層は P1 で予約されたコア機構であり、Pα は
  その具体化である(新ドメイン追加が既存ソルバ変更を要求しない、の例外ではない)」と
  位置づけを明記する。

**C-4(固定 dt × 時間加速の矛盾)** — 方針を決めて Pα 前の設計課題として登録する:

- 時間加速は World の dt を変えずに実現する: (i) 通常レンジは「1 描画フレームに複数
  物理ステップ」(dt = 1/120 固定のまま倍速)、(ii) 天体レンジ(D34)は局所物理を持たない
  **天体専用レジーム**に切り替え、独立時間軸(~0.1 日刻みのシンプレクティック積分)で進める。
  どちらも固定 dt 規約とリプレイ決定論(入力列に加速率変更イベントを記録)を破らない。
- レジーム切替(D37 再突入のように両時間軸が同時に生きる遷移)は、切替時刻の量子化
  (World ステップ境界のみ)・状態受け渡し(天体状態→ローカル初期条件の写像)・巻き戻し可否を
  定めた**切替プロトコル**として scale-ladder §2.3 に新設する。Pα 着手前に設計完了、
  D34/D37 の合格基準に「切替を跨ぐリプレイ一致」を追加する。

**C-5(CCD が Phase 5 送り)** — 弾丸級を持つ早期デモの品質欠陥を防ぐため分割する:

- **最小 CCD**(高速物体に限定した speculative contact または ray-cast bullet 方式)を
  P1 に前倒しする。対象は D2 弾道・ニュートンのゆりかご級の「単純形状・高速」ケースのみ。
- フル CCD(回転を含む一般形状の conservative advancement)は現行どおり Phase 5 に残す。
- phases.md P1 の完了条件と 10-mechanics/02 の該当節にこの分割を明記する。

**C-6(弱結合の 1 ステップ遅れ)** — 決定論と両立する形で規則を先に決める:

- sub-iteration 回数は「実行時の収束判定」ではなく**状態からの決定的関数**
  (例: 剛性指標 = 結合係数×dt/慣性 の閾値表による段階選択、上限固定)で算出する規約を
  coupling-matrix に追加する。これにより反復数もリプレイ可能になる。
- stiff な組(無慣性ロータ×回路、軽剛体×解像流体)の検出テスト(発振・発散の既知ケース)を
  P4 のテストに追加し、「兆候が出たら」を「このテストが Red になったら」に置き換える。

### D. 過剰設計への対策 — 独立トラック化

決定 1 によりドメイン削除はしない。代わりに工程を**コアトラック**と**独立トラック**に
分離する:

- **コアトラック**(クリティカルパス): math → P1(剛体)→ P2(流体格子)→ P3(SPH・熱)→
  P4(回路・光学・車両)→ 統合(D18 級の多ドメインシーン)。「部屋〜街区で遊ぶ」体験の
  最短経路であり、各ウェーブ末にデモをリリースする(B-3)。
- **独立トラック**: 量子(TDSE)、FDTD、イジング/MC、1PN 相対論、スペクトルパストレ(旧
  Phase D)。それぞれ専用デモ数本のための独立ソルバであり、コアの API(Solver トレイト・
  結合行列)にのみ依存するため、コアと並行にも後回しにも実行できる。
- ゴール定義(全テスト緑・全デモ D1–D43 合格)は変更しないが、**マイルストーンを分割**する:
  「コア完了」(コアトラックの全デモ合格)を第一級のマイルストーンとして新設し、独立トラックは
  その後(または余力での並行)とする。これによりレビューの言う「ビジョンへの忠誠が主戦場への
  到達を遅らせる」構造を、ビジョンを削らずに解消する。
- phases.md のウェーブ表・Phase C/D をこの 2 トラック構成に書き換える。

### E. 不足設計への対策

- **UI/フロントエンド**: 設計文書 `docs/23-frontend/01-demo-shell.md`(仮)を新設し、

  近似バッジ)、(ii) 予測→実験ワークフローの画面遷移、(iii) 回路エディタ(D19)の配線モデルと
  UI、(iv) シーン編集・スポーン、を設計する。P1 着手前にデモシェル部分を確定する
  (D1–D10 のリリースに必要なため)。README の文書一覧にディレクトリ行を追加する。
- **発散時巻き戻しのコスト**: architecture §5 に、スナップショットの頻度(既定: 1 s 間隔の
  リングバッファ N 面)・対象状態のサイズ見積り式(64³ 格子 + SPH 2×10⁴ の具体値)・メモリ
  予算(例: 上限 256 MB、超過時は間隔を自動延伸)を追記する。
- **wasm 配布戦略**: 05-rust-wasm-platform に配布節を追加する — 既定はシングルスレッド

### R. 第 2 回レビューの新規指摘への対策

**R-1(改訂ループが回っていない)** — 本書がループの第 1 回転である。運用ルールとして
次を phases.md 横断ルールに明文化する:

- レビュー指摘への対応は「対策方針書(本書の形式)→ 設計文書改訂 PR」の 2 段階で行う。
  レビューを受領したまま設計を 1 行も改訂せずに次のゲート(実装開始等)へ進むことを禁じる。
- 本書の承認後、[§5](#5-後続の設計文書改訂-pr-の分割案) の改訂 PR 群が完了して初めて
  vision §4 の実装開始ゲートを通過できる。

**R-2(E2 サイクロトロンが合格不能)** — 指摘は正しい(semi-implicit Euler は磁場回転で
速さが系統的に増大する)。対策:

- **Boris pusher** を integrators.md のカタログ(§1 一覧表・§2 定義・§5 トレイト)に追加
  する。Boris は磁場回転を厳密にノルム保存し、E2 の「速さ一定 abs 1e-9」を構造的に満たせる
  標準手法である。適用範囲は「電磁場中の荷電粒子・点質量」(剛体接触とは混ぜない)。
- analytic-tests の E2 行に担当積分器(Boris)を明記し、13-electromagnetism/01・05 に
  荷電粒子の運動積分は Boris を用いる旨を追記する。
- **テスト表の実装可能性監査**: E2 と同型の仕様バグ(R-5, R-6 も同種)を事前に洗い出すため、
  テスト表全行(78 本)に「担当ソルバ・担当積分器・想定実行時間・CI 層(B-2 の 3 階層)」の
  列を追加する監査パスを、該当ウェーブ着手前ではなく**改訂 PR 群の一部として一括実施**する。
  埋められない行(道具が無い・時間が乗らない)がその場で顕在化する。

**R-3(立位保持が研究級課題)** — スコープ外宣言(§3.1)と受け入れ基準(§7)の矛盾を、
基準の段階化で解消する:

- P4 の受け入れ基準を達成可能なものに置換する: 「関節 PD による静的姿勢維持 — 外乱なしで
  座位・しゃがみ姿勢を 60 s 維持(転倒しない)」+「10 N プッシュでラグドールへ遷移し、
  非物理的な発散をしない」。これは関節 PD と接触ソルバの結合が正しく動くことの検証であり、
  バランス制御を含まない。
- 「立位保持 60 s + 外乱復元」は倒立振子/ZMP 級の制御を要する**独立トラックの拡張課題**
  (エンティティ層 §3.1 の宣言どおりエンジン外)へ移し、フェーズ完了条件から外す。
- entity-layer §7・§8 と phases.md P4 を併せて改訂する。P4 は全ウェーブ中最重量との指摘も
  踏まえ、改訂時に P4 の負荷(回路・光学・WCSPH・車両・ランジュバン)の一部(例: 光学)を
  P5 へ送る再配分も検討する。

**R-4(性能予算が加算で破綻)** — 予算表に**同時アクティブ構成規則**を追加する:

- シーンを 3 クラスに分類し、クラス別に予算を定義する(05 §5 改訂):
  - **クラス S**(単一ドメイン): 現行のワークロード別予算をそのまま適用。
  - **クラス M**(2–3 ドメイン結合、例 D18): 物理合計 10 ms 以内に収まる構成のみ
    リアルタイム保証。構成例(剛体 3 + 格子流体 4 + 熱 1 + 結合 1 ≈ 9 ms)を表で示す。
  - **クラス L**(統合シナリオ、例 D37・Phase C 統合テスト): リアルタイム保証の対象外と
    明文化し、「規模を落とす(格子解像度・粒子数のプリセット)か、スロー再生(物理時間 <
    実時間)で正確に計算する」の 2 択と既定(スロー再生)を定める。vision §5.4「静かに精度を
    落とすことを禁じる」とはスロー再生既定が整合する。
- 予算 CI(06 §5)は「全シーン警告のみ」をやめ、**クラス S/M の代表シーンのみハードゲート**
  (予算超過 = fail)、クラス L は計測・可視化のみ、に改訂する。警告常態化→無視の経路を断つ。

**R-5(イジング S7/S8 の手段が工程の後ろ)** — 順序と規模の両方を直す:

- Wolff クラスタ法を「オプション・その後」から **S7/S8 を担うウェーブ(P5)の必須実装**に
  昇格する(15-statistical/04 改訂)。臨界域 L=256 は単スピン Metropolis では CI 時間に
  乗らないという指摘をそのまま採用理由とする。
- 併せて CI 階層化(B-2)を適用する: PR ゲートは L=64 の縮約版(Tc rel を緩めた基準)、
  L=256 のフル基準はナイトリー/週次層で走らせる。analytic-tests S7/S8 に層と縮約基準を
  明記する。

**R-6(F11 カルマン渦が検証モードで出ない恐れ)** — 実装前に事実を確認してから基準を
確定する:

- P2(流体格子)着手時の実装設計スパイク(A-2)の項目として「64³・渦度強化オフ・
  semi-Lagrangian で渦離脱が自発的に立ち上がるか」の数値実験を登録する。
- 結果に応じて F11 の合格条件を確定する: 立ち上がるならそのまま、立ち上がらないなら
  (i) 検証モードでも渦度強化オンを許容(強化係数を合格条件に記録)、または (ii) 解像度・
  レイノルズ数指定の変更、のいずれかに analytic-tests F11 と 11-fluid/02 §4.5 を改訂する。
  「合格基準が現象を消す」状態のまま Phase 表に残さない。

## 結語

初回レビューの結論「この設計・この工程では到達しない」に対する本書の回答は、「設計
(物理の中身とスコープ)は維持し、工程と横断機構の未決事項を直す」である。最重大の
構造問題(Phase A の BDUF、全戦線同時進行)はウェーブ局所 TDD と 2 トラック分離で解体し、
実装時に手戻りを強いる未決事項(位置表現・決定論水準・時間加速・CCD)は期限つきの
前倒し決定に変換し、仕様として壊れているテスト行(E2, S7/S8, F11, 立位保持)は個別に修理
する。第 2 回レビューが最重大とした改訂ループの不在(R-1)には、本書とその後続改訂 PR 群
(§5)がループの実績第 1 号として応答する。
