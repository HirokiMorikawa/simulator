# PR #1 設計レビュー — ゴール到達性の観点から

対象: PR #1「物理エンジン設計書一式: 物理全域をカバーする現実写像シミュレータの設計」
(53 文書・約 6,500 行、head: `claude/physics-engine-design-j89gmq` @ a775522)

レビュー観点: プロジェクトの目的(現実の写像を**検証して遊ぶ**シミュレータ)に照らし、
この設計でゴールまで問題なく到達できるか。問題点の指摘と、このまま開発を進めた場合に
生じる障害の予測を行う。対策の検討は依頼により含めない。

---

## 総評

個々の文書の質は高い。スケールの階梯(有効理論の階層 + 明示的結合)という中心思想は
物理的に正しく、決定論の敵の列挙(determinism-replay §2)、二重計上の排他規則
(coupling-matrix §2.2)、許容誤差つき解析解テスト表(analytic-tests)などは、
この種のプロジェクトの設計として上位に属する。

**しかしゴール到達性を問うなら、答えは「この設計・この工程では到達しない」である。**
致命傷は物理の中身ではなく、規模の見積り不在と工程の構造にある。

---

## A. 規模と自己評価の問題(最重大)

### A-1. 工数・期間・体制の記述が設計書群のどこにも存在しない

全文書を横断検索したが、人月・期間・チーム規模への言及はゼロである。この設計が要求する
実装は、Rapier(剛体)+ Stable Fluids/SPH(流体)+ SPICE(回路 MNA)+ Meep(FDTD)+
TDSE ソルバ + MD/イジング + REBOUND(N 体・WHFast)+ PBRT(スペクトルパストレ)の
**総和**であり、各々が単独で数年級の OSS プロジェクトに相当する。

ゴールの完了条件は「全テスト緑・全デモ D1–D43 合格」と数値で固定されている一方、
そこまでの距離を測る材料が設計内に無い。到達可能性の議論が構造的に不可能な設計書である。

### A-2. 「実装直前レベル」は過大な自己評価

vision §4 は「実装フェーズではこれらを転記・翻訳するだけで着手できる」と主張するが、
各ドメイン文書は平均 ~130 行であり、アルゴリズム選定と骨子としては優秀でも、実装の難所
— 接触マニフォールド生成の退化ケース、SPH の境界粒子とテンション不安定、MNA の非線形素子
(ダイオード)の収束、マルチグリッドの不規則境界処理 — はすべて圧縮されている。

実装時の設計乖離は確実であり、ロードマップの規約「実装が設計から乖離したら設計書を先に
改訂する」(roadmap 横断ルール 5)により、**52 文書の恒常的な二重メンテ税**が発生する。
この規約が守られなくなった時点で、設計書群は現実を写さない文書に転落する。

## B. ロードマップ(TDD)の構造問題

### B-1. Phase A「全ドメインの全テスト先行」は TDD ではなく Big Design Up Front

動くソルバが 1 つも無い段階で、全ドメインの型・トレイト・API に対して解析解テスト約 100 本
+ 各ドメイン §7 のユニットテスト + 保存則テストを先に書く計画である。一度も実行されていない
API に対するテストは、Phase B で実装が始まった瞬間に大量に書き直しになる。TDD の価値である
短い Red→Green フィードバックループが失われ、**数ヶ月間「全部 Red・緑ゼロ」の状態が続く**。
進捗の実感が無い期間として最も危険な工程設計であり、リスク表は「テスト量が大きい」ことしか
認識していない。

### B-2. CI ゲートの実行時間が未検討

全デモシーンの決定論 2 回実行 + 並列=逐次照合 + スナップショット再開一致 + 保存則 residual +
収束次数 4 水準 + ベンチ回帰を「必須ゲート」と宣言するが、例えば A2(二体 10⁶ 周のドリフト
測定)は 10⁸–10⁹ ステップ級で CI に乗らない。ゲートは数時間級に肥大するか、選択的に無効化
されて形骸化するかの二択になる。

### B-3. ユーザー価値の到達点が遠すぎる

「遊べる」最初の成果物は Phase 0 + P1 の後だが、その手前に Phase A 全体が横たわる。
また最も見栄えのする Phase D(パストレ)を「全物理完了後」にゲートしており、事実上
到達しない位置に置いている。

## C. 横断機構の技術的疑義(実装時に再設計を強いる箇所)

### C-1. ビット決定論が最も高価な不変条件になる

並列=逐次のビット一致(順序固定リダクション・赤黒順序付け)、超越関数の自前実装
`det_math`(それ自体が rlibm 級のサブプロジェクト)、FMA 無効化(性能犠牲)—
以後の**全最適化 PR がハッシュ一致テストと戦う開発税**になる。GPU は既に「許容誤差内」へ
後退しており、「ビット決定論」と「性能 3 本柱の同時適用」は恒常的に衝突する二枚看板である。

### C-2. エネルギー台帳の恒等式は実装すると恒真式に堕ちる

恒等式(architecture §1.1.2 (2))の要 $R_{numerical}$(semi-Lagrangian 移流散逸・
Baumgarte 偽仕事・splitting 誤差)は独立に計算できる量ではなく、実際にはステップ前後の
エネルギー差分 = 残差として定義するしかない。すると恒等式は自動的に成立し、バグ検出力は
「項別閾値の手動調整」に依存する。閾値が厳しければ flaky な CI、緩ければ無意味 —
設計が約束する「不明なエネルギー漏れ・湧きの常時検出」はこの形では実現しない。

### C-3. floating origin / フレーム階層が約 15 行しかない

空間上限を 10¹² m に広げた設計の要(scale-ladder §2.2)でありながら、剛体がフレームを
跨ぐ瞬間の接触・拘束の扱い、非慣性項(コリオリ)の台帳計上、跨ぎの決定論、そもそも
**位置をどのフレームで保持するか**という力学コアのデータ設計が未定義である。
これは Pα ウェーブ(最終盤)で最初のウェーブ(P1 力学)の土台改修を要求する配置であり、
「新ドメイン追加は既存ソルバの変更を要求しない」(architecture §4)という拡張原則が
自分の設計内で最初に破られる。

### C-4. 固定 dt と時間加速の矛盾が未解決

World 時刻は固定 dt = 1/120 s・可変 dt 厳禁と繰り返し宣言される一方、D34 太陽系儀は
「時間加速スライダー」を持ち、天体は独立時間軸(~0.1 日刻み)で進む。1 フレームで数日
進める操作が固定 dt の決定論とどう両立するのか、再突入のように天体時間と局所物理時間が
同時に生きるシーンのレジーム切替の具体機構(切替時の状態受け渡し・イベント時刻の量子化・
巻き戻しの有無)は、宣言のみで設計が無い。

### C-5. CCD(連続衝突検出)が Phase 5 送り

1/120 s では弾丸・高速衝突物はトンネリングする(collision-detection §に自認あり)。
D2 弾道・ニュートンのゆりかご・車両衝突は早期〜中期ウェーブにあるのに対策が最後尾 —
早期デモの品質欠陥として顕在化する。

### C-6. 弱結合の 1 ステップ遅れ

モーター結合は pre(電気→トルク)と post(ω→逆起電力)がステップを跨いで作用し、
stiff な組(無慣性ロータ×回路、軽剛体×解像流体)は設計自身が要注意と認めつつ、
sub-iteration 化の判断は「兆候が出たら」であり、反復数の決定的算出(決定論規約)との
整合は未設計である。

## D. 過剰設計(目的への寄与が薄い最大級の投資)

- **スペクトル・パストレーシング**(hero wavelength・Sobol・参加媒質・コースティクス)は
  PBRT 一冊分の別プロジェクトである。「検証して遊ぶ」体験は Three.js プレビューで成立して
  おり、目的への寄与が最も薄い場所に最大級のサブプロジェクトが接ぎ木されている。
- **量子・FDTD・イジング・1PN 相対論**は、各々が専用シナリオデモ数本のためだけの独立ソルバ
  実装(4 クレート)である。「物理の全域」というビジョンへの忠誠が、主戦場(部屋〜街区で
  遊ぶ)への到達を直接遅らせる。
- 一般シーンで定義困難な `entropy_estimate()`、Strang 分割の精度モード、GPU バックエンドの
  先行設計など、YAGNI 群が随所にある。

## E. 不足設計

- **UI / フロントエンドがほぼ白紙**。「遊ぶ」の実体は 43 本のデモ UI、自由配線の回路エディタ
  (D19)、予測→実験ワークフロー、シーン編集だが、demo 層は「実装フェーズで詳細化」の
  1 節のみ。プロダクトの半分 — ユーザーが触る側 — が設計外に置かれている。
- **発散時の巻き戻し**がデフォルト動作とされる(architecture §5)が、スナップショット頻度と
  メモリコスト(64³ 格子 + SPH 2 万粒子の全状態)は未検討。
- **wasm 性能予算が楽観的**。64³ の MG-PCG を 4 ms 等はシングルスレッド wasm では厳しい。
  wasm-bindgen-rayon は SharedArrayBuffer + COOP/COEP を要求し、静的ホスティングでの配布
  制約になるが、配布戦略の記述が無い。

## このまま進めた場合の障害予測(時系列)

1. **Phase A**: 数週間〜数ヶ月、緑ゼロのテストだけを書き続ける。Phase B 開始直後に API 変更で
   大規模なテスト書き直しが発生する。
2. **P1–P3**: Box2D 系・Stable Fluids の写経で進むが、最初の泥沼は物理ではなく決定論 CI
   (並列=逐次一致、det_math 整備)になる。
3. **P4–P5**: 専門性の壁(MNA 非線形収束・PML・SPH 安定化)で速度が急落する。設計書と実装の
   乖離が積み上がり、「設計書先行改訂」ルールが放棄され、設計書群が現実を写さなくなる。
4. **Pα**: floating origin が力学コアのデータ設計改修を要求し、「既存ソルバを変更しない拡張」
   原則が崩壊する。時間加速 × 固定 dt の矛盾がここで表面化する。
5. **Phase D**: 到達しない。
6. **全期間**: CI(決定論・保存則・ベンチ)の維持コストが実装を上回り、ゲートの選択的無効化が
   始まり、中核価値である決定論が静かに劣化する。

---

## 結語

この設計書群は「物理の百科事典としての正しさ」には合格するが、「有限の資源でゴールに到達する
工程」としては、規模の自己認識を欠いたまま全戦線同時進行を命じており、そのまま進めれば
Phase B 中盤で停滞する構造になっている。
